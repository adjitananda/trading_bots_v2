# Глоссарий терминов UTOS

**Версия:** 1.0
**Дата обновления:** 2026-04-06
**Ответственный:** ИИ-Проектировщик

---

## 1. Уровни тестирования

| Термин | Определение |
|--------|-------------|
| **L1 (Smoke-тесты)** | Быстрые автоматические тесты, которые выполняет Разработчик после каждого изменения. Время выполнения: 2-3 минуты. Проверяют, что код запускается и не падает. |
| **L2 (Регрессионные тесты)** | Полные автоматические тесты, которые выполняет Тестировщик после сдачи спринта. Время выполнения: 30-60 минут. Проверяют, что новый код не сломал существующий функционал. |
| **L3 (Ручной чек-лист)** | Ручное тестирование, которое выполняет Тестировщик. Время выполнения: 15-20 минут. Проверяет тонкие моменты (UX, читаемость логов, визуальное восприятие). |

## 2. Торговые и технические термины

| Термин | Определение |
|--------|-------------|
| **Market Regime** | Режим рынка. Определяется детектором на основе ADX, ATR и положения относительно EMA200. Возможные значения: HIGH_VOLATILITY, STRONG_UPTREND, STRONG_DOWNTREND, RANGING. |
| **Градуированные триггеры** | Система уровней тревоги: 0 (OK — нормальная работа), 1 (Warning — снизить риск), 2 (Halt — остановить торговлю по символу). |
| **reload_flag** | Поле в таблице `bot_symbols`. При установке в 1 сигнализирует боту, что нужно перезагрузить параметры стратегии из БД без перезапуска процесса. |
| **overfit_ratio** | Показатель переобучения модели. Рассчитывается как `train_sharpe / test_sharpe`. Норма: <1.1. Приемлемо: 1.1-1.5. Переобучение: >1.5. |
| **out-of-sample** | Метод проверки стратегии на данных, которые не участвовали в оптимизации. В UTOS используется разделение 80% train / 20% test. |
| **risk_multiplier** | Множитель размера позиции. Уменьшается при Warning (0.5) или Halt (0.0). Также может снижаться Volatility Guard при высокой волатильности. |

## 3. Архитектурные термины

| Термин | Определение |
|--------|-------------|
| **trading_lib** | Общая библиотека проекта. Содержит адаптеры бирж, стратегии, оптимизатор, риск-менеджмент, утилиты. Импортируется всеми ботами. |
| **bot_symbols** | Таблица в БД, связывающая бота с торговыми символами. Позволяет одному боту управлять несколькими символами с разными параметрами. |
| **CRYPTO_BOT** | Бот для торговли криптовалютами на Bybit. Может управлять несколькими символами (BTCUSDT, ETHUSDT, SOLUSD и др.). |
| **TINKOFF_BOT** | Бот для торговли акциями на Тинькофф. Учитывает торговые сессии (10:00-18:50 МСК) и выходные дни. |
| **MOEX_BOT** | Бот для торговли фьючерсами на Мосбирже. Учитывает сессии (10:00-23:50 МСК) и экспирацию контрактов. |

## 4. Роли в проекте

| Термин | Определение |
|--------|-------------|
| **Архитектор** | Пользователь. Принимает финальные решения, формулирует идеи, утверждает стратегию, исполняет действия (отправка команд, коммиты). |
| **ИИ-Проектировщик** | Управляет проектом, формирует ТЗ, координирует Разработчика и Тестировщика, принимает спринты. |
| **ИИ-Консультант** | Независимый эксперт по трейдингу. Даёт оценки, указывает на риски, участвует в стратегических совещаниях. |
| **ИИ-Разработчик** | Пишет код, запускает L1-тесты, сдаёт спринт Проектировщику. |
| **ИИ-Тестировщик** | Выполняет L2 и L3 тестирование, составляет отчёты о результатах. |

---

## 5. Процессные термины

| Термин | Определение |
|--------|-------------|
| **Спринт** | Цикл разработки, результатом которого является работающий код и отчёт о тестировании. Длительность: 1-2 недели. |
| **Эксплуатационный этап** | Период без активной разработки, только мониторинг, сбор метрик и настройка порогов. |
| **Бэклог** | Список задач в GitHub Issues, которые не вошли в текущий спринт, но могут быть реализованы в будущем. |
| **Known issues** | Список известных, но некритичных ошибок, которые принимаются вместе со спринтом и исправляются позже. |

---

## 6. Классификация ошибок

| Тип ошибки | Определение | Влияние на приёмку спринта |
|------------|-------------|----------------------------|
| **Критическая** | Не работает основной функционал, потеря данных, падение бота | Спринт не принимается |
| **Значительная** | Работает с ошибками, влияет на некоторые сценарии | Можно принять с понижением приоритета (ошибка в known issues) |
| **Косметическая** | Опечатки, некрасивые сообщения, медленный лог | Можно принять (ошибка в бэклог) |
