# Регламент работы с GitHub Issues (бэклог)

**Версия:** 1.0
**Дата обновления:** 2026-04-06
**Ответственный:** Архитектор
**Регламент обновления:** При изменении процесса или по запросу Координатора

---

## 1. Назначение

Этот документ описывает, как Архитектор ведёт бэклог проекта UTOS с помощью GitHub Issues.

**Инструмент:** https://github.com/adjitananda/trading_bots_v2/issues

---

## 2. Когда создавать Issue

- **Новая идея** от Координатора, Консультанта или самого Архитектора
- **Найденный баг** (Тестировщиком, Координатором или Архитектором)
- **Технический долг** (необходимость рефакторинга, удаления старого кода)
- **Улучшение UX** (неудобная команда, нечитаемый вывод)
- **Новый символ** для существующего бота
- **Задача из бэклога**, которая не вошла в текущий спринт

---

## 3. Как создавать Issue (ручной режим)

1. Перейти на страницу `Issues` репозитория
2. Нажать **«New issue»**
3. Заполнить поля:

| Поле | Что писать | Пример |
|------|------------|--------|
| **Title** | Краткая суть задачи | «Добавить поддержку AVAX на CRYPTO_BOT» |
| **Description** | Подробное описание (см. шаблон ниже) | |
| **Labels** | Выбрать из списка | `enhancement`, `new_symbol` |
| **Assignees** | (опционально) `Architect` | |
| **Milestone** | (опционально) номер спринта | `Sprint 9` |

### Шаблон описания Issue

```markdown
**Тип:** [новая фича / баг / улучшение UX / новый символ / техдолг]
**Приоритет:** [Критический / Высокий / Средний / Низкий]
**Источник:** [Координатор / Консультант / Тестировщик / Архитектор]

## Описание
[Чёткое описание задачи]

## Ожидаемый результат
[Что должно получиться после выполнения]

## Дополнительно
[Логи, ссылки, примеры команд, если есть]
4. Метки и их значение
Метка	Когда использовать
enhancement	Новая функциональность (фича)
bug	Ошибка, неправильное поведение
new_symbol	Добавление нового торгового инструмента
ux	Улучшение интерфейса (Telegram, вывод команд)
documentation	Задача по документации (обновить process/, prompts/)
technical_debt	Рефакторинг, удаление старого кода, миграция
5. Приоритеты
Приоритет	Описание	Когда применять
Критический	Ошибка, блокирующая торговлю или приводящая к убыткам	Немедленно
Высокий	Сильно влияет на работу, но есть обходной путь	В текущий спринт
Средний	Улучшение, которое подождёт 1-2 спринта	В следующий спринт
Низкий	Косметика, nice-to-have	В бэклог
6. Процесс работы с Issues
6.1. Планирование спринта
Архитектор просматривает открытые Issues

Отбирает задачи для спринта (с учётом приоритетов)

Переносит их в ТЗ (ссылается на номер Issue)

Назначает Issue на спринт через Milestone

6.2. Во время спринта
Если Issue взят в работу — добавить комментарий: «Взято в работу. Спринт №X»

Если Issue заблокировано — добавить комментарий с причиной

6.3. После выполнения
Разработчик или Тестировщик сообщает Архитектору о выполнении

Архитектор проверяет (через Тестировщика)

Архитектор закрывает Issue с комментарием: «Выполнено в Спринте №X. Коммит: ...»

7. Что НЕ нужно заносить в Issues
Миссию, архитектуру, роли, принципы (это в Project-passport.md)

Дорожную карту (это в Roadmap.md)

Глоссарий (это в Glossary.md)

Стандарты кодирования (это в Coding-standards.md)

Регламенты ролей (это в prompts/*.md)

8. Примеры правильно оформленных Issues

Description:

markdown

**Приоритет:** Средний
**Источник:** Координатор

## Ожидаемый результат

- Команда /crypto_metrics AVAXUSDT работает
- Параметры для оптимизации как у ETHUSDT
