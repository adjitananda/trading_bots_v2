## 1. Миссия и стратегическая цель проекта

### 1.1. Миссия

Создать полностью автономную торговую систему, которая самостоятельно торгует на множестве бирж и инструментов, непрерывно анализирует свою эффективность, оптимизирует параметры, выбирает лучшие стратегии и адаптируется к изменяющимся рыночным условиям — с человеком в роли наблюдателя и финального утверждающего.

### 1.2. Стратегическая цель (конечное состояние)

Система торгует 24/7 на криптовалютных и фондовых рынках (Bybit, Тинькофф, Мосбиржа). При ухудшении метрик (просадка, падение Sharpe) система автоматически запускает оптимизацию параметров или переключение стратегии, предлагает изменения через Telegram, после утверждения применяет их без остановки торговли. Человек контролирует, но не вмешивается в рутину.

## 2. Архитектура проекта (высокоуровневая)

### 2.1. Ключевое архитектурное решение

**Один бот = одна биржа.** Отдельные процессы для каждой биржи. Это обеспечивает изоляцию сбоев и упрощает учёт разных торговых сессий.

### 2.2. Текущие боты и биржи

| Бот | Биржа | Тип инструментов | Статус |
|-----|-------|------------------|--------|
| CRYPTO_BOT | Bybit | Криптовалютные фьючерсы | ✅ Активен |
| TINKOFF_BOT | Тинькофф | Акции | ✅ Активен |
| MOEX_BOT | Мосбиржа | Фьючерсы | ✅ Активен |
| Старые single-coin боты (btc_bot, ada_bot...) | Bybit | Криптовалютные фьючерсы | ✅ Активны (подлежат миграции) |

### 2.3. Общая библиотека `trading_lib`

Весь общий код вынесен в `trading_lib/`. Боты импортируют из неё адаптеры, стратегии, оптимизатор, риск-менеджмент и утилиты.

**Структура `trading_lib`:**
- `exchanges/` — адаптеры бирж (Bybit, Tinkoff, Moex)
- `strategies/` — торговые стратегии (MA, Bollinger, SuperTrend)
- `optimizer/` — автооптимизация (Optuna, метрики, триггеры)
- `regime/` — фильтр режимов рынка (ADX, ATR, волатильность)
- `utils/` — общие утилиты (БД, валидация символов, время)
- `telegram/` — уведомления (без команд, только отправка)

### 2.4. База данных

Единая MySQL БД `trading_bots_v2` для всех ботов.

**Основные таблицы:**
- `bots` — описание ботов (общее)
- `bot_symbols` — связь бот → символ, параметры стратегии
- `trades`, `orders` — сделки и ордера
- `optimization_history` — история оптимизаций
- `bot_performance_metrics` — ежедневные метрики
- `market_regime_log` — логи режимов рынка
- `trigger_log` — логи срабатываний триггеров

## 3. Иерархия и роли в проекте

### 3.1. Иерархия подчинения
Координатор (стратегия, идеи, финальные решения)
│
└── Архитектор (управление проектом, ТЗ, приёмка)
│
├── Разработчик (исполнение)
├── Тестировщик (контроль качества)
│
└── Консультант (независимый эксперт, совещательный голос)

text

### 3.2. Роли и зоны ответственности

| Роль | Ответственность | Кому подчиняется |
|------|----------------|------------------|
| **Координатор** | Формулирует идеи, принимает финальные решения, утверждает стратегию, исполняет действия (отправка команд, коммиты) | — |
| **Архитектор** | Управление проектом, формирование ТЗ, координация ролей, приёмка спринтов | Координатор |
| **Консультант** | Экспертиза по трейдингу, оценка рисков, рекомендации, участие в стратегических совещаниях | Координатор (совещательно) |
| **Разработчик** | Написание кода, L1-тесты, сдача спринта | Архитектор |
| **Тестировщик** | L2 и L3 тестирование, отчёт о результатах | Архитектор |

### 3.3. Принцип "Координатор — единственный исполнитель"

Только Координатор имеет физический доступ к серверу, репозиторию, Telegram и БД. Все остальные роли дают инструкции и код, но не исполняют действия автоматически.

## 4. Фундаментальные принципы и ограничения

### 4.1. Принятые архитектурные решения

| Принцип | Обоснование |
|---------|-------------|
| Один бот — одна биржа | Изоляция сбоев, учёт разных торговых сессий, упрощение отладки |
| Multi-coin через `bot_symbols` | Один процесс управляет несколькими символами с разными параметрами |
| Общая библиотека `trading_lib` | Единый код для всех ботов, избегание дублирования |
| Telegram с префиксами | `/crypto_metrics`, `/tinkoff_metrics`, `/moex_metrics` |

### 4.2. Торговые принципы (от Консультанта)

| Принцип | Суть |
|---------|------|
| Без фильтра рынка — никуда | Автооптимизация опасна без определения режима (тренд/флэт/высокая волатильность) |
| Out-of-sample обязателен | 20% данных для проверки, порог overfit_ratio = 1.5 |
| Составной критерий вместо чистого Sharpe | Sharpe + Calmar + Profit Factor + мин. количество сделок |
| Градуированные триггеры (0/1/2) | Warning (снижение риска) → Halt (остановка), а не бинарное срабатывание |
| HIGH_VOLATILITY запрещает всё | Приоритет режима — самый высокий |

### 4.3. Запрещённые подходы (veto)

| Запрет | Причина |
|--------|---------|
| Grid Trading на старте | Убивает во волатильность, требует отдельного Risk Manager |
| Ensemble стратегии на старте | Слишком сложно, гарантированное переобучение |
| Универсальный бот на все биржи | "Эффект домино" — один сбой останавливает всё |
| Prediction Module в V1 | ML в трейдинге — отдельная наука, риск переобучения. Отложено до V2.0 |
| Sharpe как единственный критерий | Не штрафует за просадки и редкие сделки |

### 4.4. Приоритеты развития (на V1)

1. Стабильность и реальная торговля (приоритет)
2. Миграция single-coin ботов на CRYPTO_BOT
3. Расширение инструментов (новые символы) — в бэклог
4. Prediction Module — V2.0
5. Финам как брокер — резерв

## 5. Процесс разработки и тестирования

### 5.1. Уровни тестирования (L1/L2/L3)

| Уровень | Название | Кто выполняет | Что проверяет |
|---------|----------|---------------|---------------|
| **L1** | Smoke-тесты | Разработчик | Код запускается, не падает (2-3 минуты) |
| **L2** | Регрессионные тесты | Тестировщик | Новый код не сломал старый (30-60 минут) |
| **L3** | Ручной чек-лист | Тестировщик | Тонкие моменты, UX, логи (15-20 минут) |

### 5.2. Классификация ошибок при приёмке спринта

| Тип ошибки | Можно ли принять спринт | Действие |
|------------|------------------------|----------|
| **Критическая** (не работает основной функционал, потеря данных) | ❌ НЕТ | Спринт не принимается, ошибка исправляется |
| **Значительная** (работает с ошибками, влияет на сценарии) | ⚠️ ДА, с понижением приоритета | Ошибка в `known_issues.txt`, исправление в следующем спринте |
| **Косметическая** (опечатки, некрасивые сообщения) | ✅ ДА | Ошибка в бэклог |

### 5.3. Алгоритм работы над спринтом

1. **Архитектор** получает задачу от **Координатора**
2. **Архитектор** консультируется с **Консультантом** (при необходимости)
3. **Архитектор** формирует единое ТЗ (разделы: Разработка + Тестирование)
4. **Архитектор** передаёт ТЗ **Разработчику**
5. **Разработчик** пишет код, запускает L1, сдаёт спринт
6. **Архитектор** передаёт тот же ТЗ (раздел тестирования) **Тестировщику**
7. **Тестировщик** выполняет L2 и L3, составляет отчёт
8. **Архитектор** принимает решение:
   - ✅ Принять
   - ❌ Отправить на доработку (цикл повторяется)

## 7. Отчёты и документация

### 7.1. Еженедельные отчёты

- **Формат:** Telegram (Координатор отправляет) + файл `reports/weekly_YYYY-MM-DD.md` в репозитории
- **Содержание:** статус ботов, метрики, сработавшие триггеры, ошибки, план
- **Кто готовит:** Архитектор (даёт инструкцию Координатору)

### 7.2. Бэклог (GitHub Issues)

**Инструмент:** GitHub Issues репозитория: https://github.com/adjitananda/trading_bots_v2/issues

**Назначение:** Хранение и отслеживание задач, которые не вошли в текущий спринт (новые идеи, баги, улучшения, технический долг).

**Кто создаёт Issue:**
- Координатор (озвучивает идею) → Архитектор (оформляет Issue)
- Архитектор (самостоятельно, по результатам анализа)
- Тестировщик (найденный баг) → Архитектор (оформляет Issue)

**Кто ведёт бэклог (добавляет метки, приоритеты, закрывает):** Архитектор

**Стандартные метки:**
- `enhancement` — новая функциональность
- `bug` — ошибка
- `new_symbol` — добавление нового торгового инструмента
- `ux` — улучшение удобства использования
- `documentation` — задачи по документации
- `technical_debt` — технический долг (рефакторинг, удаление старого кода)

**Приоритет (в заголовке или описании):** Критический / Высокий / Средний / Низкий

**Процесс:**
1. Issue создаётся (с метками, приоритетом, описанием)
2. Архитектор анализирует и при планировании спринта назначает Issue в ТЗ
3. После выполнения задачи Issue закрывается с комментарием

> **Важно:** Issues не дублируют документы `process/` и `prompts/`. Это рабочий инструмент для конкретных задач, а не архив документации.

**Автоматизация:** В планах — создание скрипта `scripts/finalize_sprint.py` для автоматического коммита и создания Issue по итогам спринта (ориентировочно Спринт №9).
